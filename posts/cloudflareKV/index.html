📚 Cloudflare 图床缓存优化实战总结

🎯 项目背景

项目：Cloudflare-ImgBed（基于 Cloudflare Pages + Workers + KV + telegram 的自建图床）
问题：免费 KV 存储的 10 万次/日读取额度很快用完，需要付费
目标：通过缓存优化，减少 80% 以上的 KV 读取，完全使用免费额度

🔍 问题分析

  1. 每次图片访问流程：Worker → 验证 → KV读取 → telegram读取 → 返回
  2. 重复访问浪费：同一张图片被多次访问时，每次都要走完整流程
  3. 核心问题：缺少缓存层，无法利用图片的静态特性

💡 解决方案

核心思路：三级缓存策略

技术实现：Cloudflare Cache API

🛠️ 实施步骤

步骤1：确定文件位置

步骤2：添加缓存中间件

在 /functions/_middleware.js 中添加缓存逻辑，关键点：

  • 只缓存图片请求（.jpg, .png, .webp等）
  • 设置合适的缓存时间（7-30天）
  • 添加调试日志和响应头

步骤3：测试验证

  1. 打开浏览器，按 F12 进入开发者工具
  2. 选择 Network 标签页
  3. 勾选 ** 禁用缓存**（避免浏览器缓存干扰）
  4. 访问你的图片URL：https://你的域名/file/xxx.jpg

在Network面板点击请求，查看：

步骤4：监控效果

在 Cloudflare Dashboard 查看：

  1. KV 读取数：应该大幅下降
  2. Worker 请求数：缓存命中时不计数
  3. 响应头：确认 cf-cache-status: HIT

📊 效果对比

优化前：

  • KV读取：每次访问 +1
  • 响应时间：200-500ms
  • 免费额度：很快用完

优化后：

  • KV读取：只有第一次访问 +1，后续 +0
  • 响应时间：缓存命中时 10-50ms（提升10倍）
  • 免费额度：完全够用

🚀 性能提升

  1. 速度：首次加载 300ms → 后续加载 30ms
  2. 成本：可能超额度 → 完全免费
  3. 可扩展性：支持更高并发

⚠️ 注意事项

1. 缓存时间选择

  • 普通图片：7-30天
  • 头像/Logo：30-90天
  • 临时图片：1-24小时

2. 缓存清除

3. 缓存键设计

  • 使用完整URL作为缓存键
  • 注意查询参数的影响
  • 考虑不同尺寸版本的缓存

📈 监控指标

关键指标：

  1. 缓存命中率 = HIT次数 / 总请求数（目标 > 85%）
  2. KV读取节省率 = (1 - 优化后读取/优化前读取) × 100%
  3. 响应时间P95：缓存命中时应 < 50ms

Cloudflare Dashboard 查看位置：

  1. Workers & Pages → 你的项目 → 指标
  2. Workers KV → KV实例 → 使用情况
  3. 分析 → 性能
